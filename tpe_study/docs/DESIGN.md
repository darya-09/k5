# DESIGN — что где, какие эксперименты, какие алгоритмы, в каком порядке и почему

Этот документ объясняет архитектуру `tpe_study` так, чтобы по нему можно было воспроизвести
и защитить работу, не читая весь код.

## 0. Цель
Сравнить **способы изменения TPE** (Tree-structured Parzen Estimator) на 2D benchmark-функциях
честно — так, чтобы каждое сравнение «базовый TPE vs модификация» меняло **ровно один фактор**.

## 1. Почему отдельная чистая реализация, а не исходные ноутбуки
В исходных ноутбуках были три проблемы корректности, которые мешают сравнению:
1. **w(x)**: repo-`TPEOptimizer` передаёт в `weight_fn` среднее координат `(x0+x1)/2`, а ноутбук
   считал градиент в точке `(mean, 0)` — т.е. **не тот градиент**.
2. **gTPE**: была отдельной реализацией → отличалась от repo-TPE сразу по многим осям (не ablation).
3. **Нормализация при шуме**: seed шума зависел от `scale`, поэтому raw и norm видели **разный шум**.

Здесь всё это исправлено: одна реализация TPE с флагами, истинный градиент в реальной точке,
единый шум для raw/norm. Зависимости — только `numpy/scipy/optuna/pandas/matplotlib` (без Drive).

## 2. Структура каталога
```
tpe_study/
  run.py                      # ГЛАВНЫЙ запуск: считает всё, пишет таблицы и графики
  configs/default.json        # единый конфиг (seeds, бюджет, сетка) — один источник правды
  src/tpe_study/
    functions.py              # benchmark-функции + аналитические градиенты + минимумы + bounds
    objective.py              # что ВИДИТ оптимизатор: scale(raw|norm) и data(clean|noisy_y) + шум
    tpe.py                    # ЧИСТЫЙ TPE с флагами-модификациями (+ 1D KDE, + мини-GP)
    baselines.py              # random search и Optuna TPE (внешние референсы)
    metrics.py                # метрики по RAW CLEAN: dist_y, dist_x, шаги до порога, агрегация
    experiment.py             # реестр алгоритмов + прогон ячейки сетки по seeds
    plots.py                  # кривые сходимости и карты выбора точек
  notebooks/                  # Colab-ноутбуки: обзор, запуск, анализ результатов
  results/tables/*.csv        # ТАБЛИЧНЫЕ РЕЗУЛЬТАТЫ (после запуска)
  results/figures/*.png       # графики (после запуска)
  docs/DESIGN.md, RESULTS.md  # этот файл и файл результатов
```

## 3. Откуда что берётся (поток данных)
```
functions.py (f, ∇f, x*, bounds)
        │
        ▼
objective.py  ── строит то, что видит оптимизатор:
        │        value = f(x) [/f_max если norm] [+N(0,σ) если noisy]
        │        grad  = ∇f(x) [/f_max если norm]   (шум только в value)
        ▼
experiment.py ── для каждой ячейки (function, scale, data, algorithm) гоняет по всем seeds:
        │          algorithm ∈ {random, tpe, tpe_gradw, tpe_gp, tpe_gradw_gp, optuna}
        ▼
tpe.py / baselines.py ── возвращают x_history, y_history (наблюдаемые)
        ▼
metrics.py ── ВСЕГДА пересчитывает качество по RAW CLEAN f:
        │        best-so-far по наблюдаемому y → dist_y(t)=|f(x_best)-f*|, dist_x(t)=||x_best-x*||
        ▼
run.py ── агрегирует по seeds, пишет CSV (all_results, summary, ablation) и PNG.
```

## 4. Алгоритмы (что именно сравнивается)
Все, кроме `random`/`optuna`, — это ОДИН класс `TPE` с одним изменённым флагом:

| algorithm | флаг | что проверяет |
|---|---|---|
| `random` | — | нижняя граница |
| `tpe` | базовый | контрольная точка (rank-based l(x)/g(x)) |
| `tpe_gradw` | `gradient_weight=True` | **фактор: градиентное взвешивание** наблюдений KDE по 1/‖∇f‖ (в реальной точке) |
| `tpe_gp` | `gp_rerank=True` | **фактор: GP-переранжирование** кандидатов |
| `tpe_gradw_gp` | оба | взаимодействие (это «gTPE») |
| `optuna` | — | внешний зрелый референс |

**Нормализация** — это фактор `scale` (свойство задачи, общий для всех алгоритмов).
Сравнение «нужна ли нормализация» = `tpe@raw` против `tpe@norm`.

## 5. Сетка эксперимента и порядок
Полностью факторная: `функция(4) × scale(2) × data(2) × algorithm(6) × seed(N)`.
Порядок в `run.py`: функции → scale → data → алгоритмы → seeds. Для каждой тройки
`(function, scale, data)` строятся графики по всем алгоритмам сразу.

## 6. Корректность сравнения (ключевые принципы)
1. **Общие случайные числа:** objective с одним `(function, data, seed)` даёт один и тот же шум
   всем алгоритмам и обеим scale → парные сравнения.
2. **Единые гиперпараметры:** n_init, gamma, n_candidates, max_evals — из одного конфига для всех.
3. **Оценка по raw clean:** что бы ни видел оптимизатор, метрика честная.
4. **Единый порог** успеха на функцию (по dist_y).
5. **Детерминизм:** `PYTHONHASHSEED=0` + фиксированные seeds + фиксированный seed оценки f_max.

## 7. Что считать ablation, а что нет
- `tpe` vs `tpe_gradw` / `tpe_gp` / `tpe_gradw_gp` — корректные ablation (один фактор).
- `tpe@raw` vs `tpe@norm` — корректный ablation нормализации.
- `optuna`, `random` — внешний контекст, не ablation (другие реализации).

## 8. Известные допущения (честно)
- Градиент **аналитический и точный** (оракул) — это исследование «помогает ли точный градиент»,
  а не готовый black-box HPO. Чтобы приблизить к реальности, в `objective.py` легко добавить шум
  в градиент (сейчас шум только в значении).
- 2D и умеренный один уровень шума на функцию; для статьи стоит добавить sensitivity по σ и размерности.
- GP используется только для переранжирования кандидатов (не полноценный BO).

## 9. Как запустить
```bash
cd tpe_study
PYTHONHASHSEED=0 python run.py                # полный прогон (configs/default.json)
PYTHONHASHSEED=0 python run.py --quick        # быстрая проверка (мало seeds)
```
Результаты появятся в `results/tables/*.csv` и `results/figures/*.png`,
сводка с числами — в `docs/RESULTS.md`.
