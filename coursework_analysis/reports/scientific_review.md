# Научная корректность: ревью

Раздел отвечает на вопросы из ТЗ (п.8). Чётко разделены: ✅ что подтверждено кодом/данными,
⚠️ технические проблемы, 🔴 научные проблемы.

## 8.1 Проверяет ли код заявленную цель?
Цель — «сравнить способы изменения ТПЕ». Способов реально три:
1. **Нормализация objective** (fin_3) — ✅ проверяет, но результат: для рангового TPE это **инвариантность** (no-op на clean).
   То есть как «способ модификации» нормализация для TPE неинформативна — и это сам по себе валидный вывод.
2. **w(x) градиентный вес** (fin_4) — проверяет, НО под вопросом корректность вызова weight_fn (см. 🔴-1).
3. **gTPE** (fin_5) — проверяет другую реализацию целиком, не изолированный фактор (🔴-2).

## 8.2 Достаточно ли экспериментов?
Частично. Есть baseline, есть три модификации, 4 функции × 2 шкалы × 2 типа данных × 30 seeds — это хороший охват.
Не хватает: (а) статистических тестов значимости различий, (б) нескольких уровней шума, (в) проверки 2D-вызова weight_fn.

## 8.3 Есть ли baseline? — ✅ Да
`random`, `no_w` (repo-TPE), `optuna` присутствуют во всех основных ноутбуках.

## 8.4 Меняется ли в каждом эксперименте ровно один фактор?
- fin_3 clean: ✅ один фактор (raw↔norm), и он оказался нейтральным.
- fin_3 noisy: 🔴 **два фактора сразу** — шкала И реализация шума (seed зависит от scale_type).
- fin_4 (no_w vs grad_*): ⚠️ формально один фактор (наличие weight_fn), общий шум по seed — но сам фактор реализован
  с дефектом (см. 🔴-1): w(x) опирается на градиент в неверной точке, поэтому это «ablation сломанного фактора».
- fin_5 (gTPE vs no_w): 🔴 **много факторов сразу** (реализация + веса + GP).

## 8.5 Эксперименты, где меняется несколько факторов
- fin_3 noisy raw vs norm (шкала + шум).
- fin_5 gTPE vs repo-TPE (реализация + grad-веса + GP + точный градиент в noisy).

## 8.6 Sensitivity analysis — ⚠️ почти нет
Единственный «sensitivity» — clean vs noisy_y при ОДНОМ уровне σ (fin_2/3/4/5). Нет варьирования σ, n_init, top/γ, n_candidates.

## 8.7 Проверка устойчивости результата
Есть дисперсия/медиана по 30 seeds (var/median колонки) — ✅ базовая устойчивость считается.
Нет: доверительных интервалов, тестов (Уилкоксон/перестановочный), поправок на множественные сравнения.

## 8.8 Технические проблемы (⚠️)
- ⚠️ **Невоспроизводимо из git:** пакет `tpe/` только в Drive; без него ничего не импортируется.
- ⚠️ **n_init=25 в fin_1** против 10 в остальных → нельзя класть в одну таблицу сравнения.
- ⚠️ **Разные пороги** success в базах (01/02) и в 03/04/05 → success_rate несравним сквозь серию.
- ⚠️ `hash(...)` в seed шума зависит от `PYTHONHASHSEED` — между сессиями шум может отличаться (нет фиксации hash seed).
- ⚠️ Optuna в fin_4/fin_5: `trial.suggest_float(name, lo, hi)` через `bounds[...][1],[2]` — порядок аргументов корректен, но стоит перепроверить имена осей x0/x1.

## 8.9 Научные проблемы (🔴)
- 🔴 **weight_fn вызывается на СРЕДНЕМ координат — ПОДТВЕРЖДЕНО по `tpe/optimizer/tpe.py:232-252`.**
  repo-TPE передаёт в `weight_fn` 1D-массив `mean(x0,x1)` по кандидатам, а ноутбук считает ∇f в точке (mean,0).
  → метод 2 **не** реализует градиентное взвешивание; его результаты нельзя трактовать как «gradient-aware TPE».
  Плюс `scale_to_range` стирает амплитуду веса (остаётся только ранг). **Это закрытый вопрос: метод 2 надо переписать.**
- 🔴 **«Оракульный» градиент:** методы 2 и 3 используют ТОЧНЫЙ ∇f benchmark-а. В реальном HPO градиента нет/он шумный. Поэтому результат — про «можно ли помочь TPE точным градиентом», а не про практичный метод.
- 🔴 **noisy raw-vs-norm спутан по шуму** (fin_3) — нельзя интерпретировать.
- 🔴 **gTPE — не ablation** — нельзя приписывать разницу одному фактору.

## 8.10 Какие выводы можно делать строго (сейчас)
1. repo-TPE > random; ≈/≥ Optuna на benchmark-наборе (01).
2. Нормализация objective не меняет поведение рангового TPE на clean (инвариантность) — **сильный, корректный вывод** (03).
3. При малом шуме базовый TPE устойчив (02).
4. gTPE не доминирует, на гладких функциях уступает repo-TPE/Optuna (05).
5. **Метод 2 (w(x)) реализован с дефектом:** weight_fn получает среднее координат, ∇f считается в точке (mean,0) →
   это не градиентное взвешивание. Строгий вывод: текущие grad_*-результаты невалидны как «gradient-aware TPE».

## 8.11 Какие выводы пока нельзя делать
- «w(x) по градиенту улучшает/ухудшает TPE» — метод считает градиент в неверной точке; нужен исправленный прогон + стат-тест.
- «нормализация помогает при шуме» — сравнение спутано.
- «gTPE лучше/хуже из-за градиента» — смешаны факторы.

## 8.12 Рекомендуемые дополнительные эксперименты
1. ✅ **Вызов weight_fn проверен** (🔴-1 закрыт): repo-TPE передаёт mean(x0,x1). **Действие:** исправить метод 2 —
   передавать в w(x) реальные 2D-точки (доработать repo-TPE или подкласс), и только потом повторять fin_4.
2. **Sensitivity по σ** (например σ×{0.5,1,2,4}) и по n_init/γ.
3. **Стат-значимость:** парный Уилкоксон no_w vs grad_* по 30 seeds + поправка Холма.
4. **Честный noisy для fin_3:** общий шум для raw/norm.
5. **Ablation для gTPE:** включать по одному (только веса; только GP; веса+GP) на той же реализации.
6. **Шумный/конечно-разностный градиент** вместо аналитического — приблизить к реальности.
